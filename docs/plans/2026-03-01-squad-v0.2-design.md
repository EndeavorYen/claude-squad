# Squad Plugin v0.2 設計文件

> Date: 2026-03-01 | Based on: Rexiano 任務實戰回饋（6 agent、2 平行工作流）

## 背景

Squad v0.1 在一次大型實戰任務中暴露了多個系統性問題。該任務部署了 6 個 agent（Alpha, Bravo, Delta, Echo, Foxtrot, Golf）跨越 UI 翻新和 Sprint 5 功能兩個平行工作流。以下是發現的核心痛點及對應的設計改進。

### 核心痛點摘要

| # | 痛點 | 影響程度 | 根本原因 |
|---|------|---------|---------|
| 1 | EXECUTE 和 VERIFY 之間缺少合併步驟 | Critical | Pipeline 假設 agent 直接在主工作區作業 |
| 2 | Worktree base commit 漂移（32 commits behind） | Critical | 未強制從 HEAD 建立分支 |
| 3 | ESLint 掃描 .claude/worktrees/ 目錄（0→86 errors） | High | 未排除 squad 工作目錄 |
| 4 | Context compaction 導致 agent 斷線 | High | 依賴 in-memory SendMessage 通訊 |
| 5 | TeamDelete 失敗阻塞 pipeline | Medium | 無容錯清理機制 |
| 6 | 共享 store 衝突（簡化版 vs 完整版） | High | Agent 看不到共享檔案現有結構 |
| 7 | 6 agent 同時運行 context 爆量 | Medium | 無分批部署策略 |
| 8 | 無前置檢查/回滾機制 | Medium | Pipeline 假設一切順利 |
| 9 | 無主動中止機制 | Low | Stop hook 只防止意外退出 |

## 設計改進

### 1. 新增 INTEGRATE 階段（P0 — Critical）

**變更：** Pipeline 從 6 階段變為 7 階段

```
RECON → PLAN → EXECUTE → INTEGRATE → VERIFY → DEBRIEF → RETRO
```

**影響的檔案：**
- `commands/squad.md` — 新增 INTEGRATE 區塊
- `skills/gate-check/SKILL.md` — 新增 INTEGRATE gate
- 新增 `skills/integration/SKILL.md`

**INTEGRATE 四階段流程：**

#### Phase 1 — 清點（Inventory）

讀取每個 agent 的輸出目錄（見改進 3）：
- `.claude/squad/outputs/{callsign}/manifest.md` — 修改檔案清單
- `.claude/squad/outputs/{callsign}/interface-changes.md` — 共享檔案變更

建立衝突矩陣：哪些檔案被多個 agent 觸及。

#### Phase 2 — 衝突偵測

對衝突矩陣中的每個共享檔案：
- 讀取各 agent 的 `interface-changes.md` 確認變更性質
- 分類：
  - **Additive（相容）**：不同 agent 新增不同欄位到同一檔案 → 可自動合併
  - **Conflicting（衝突）**：同一欄位被多個 agent 修改 → 需要人工決策
- 產出衝突報告

#### Phase 3 — 合併執行

按 dependency wave 順序（Wave 1 agent 先合併）處理每個 agent：

**Worktree agent 合併策略：**
1. 讀取 agent 的 `manifest.md` 取得新建/修改檔案清單
2. **新建的檔案**：直接從 worktree 複製到主工作區
3. **修改的獨佔檔案**（無衝突）：直接從 worktree 複製覆蓋
4. **修改的共享檔案**（有衝突）：
   - 讀取 agent 的 `interface-changes.md`
   - 用 additive merge 策略合併（新增欄位/exports/types）到主工作區版本
   - 不覆蓋、不重建，只新增

**File-boundary agent 合併策略：**
- 無需合併——已在主工作區作業
- 僅驗證 agent 沒有超出其檔案邊界

**None 隔離 agent 合併策略：**
- 複製產出檔案（docs、plans）到目標位置

#### Phase 4 — 清理

1. 刪除所有 worktree：`git worktree remove` 每個 agent worktree
2. 執行 `git worktree prune`
3. 清理 `.claude/squad/outputs/` 目錄
4. 驗證 `.claude/worktrees/` 已清空

**Gate Check：**

| Gate Level | After INTEGRATE |
|------------|----------------|
| supervised | Pause |
| standard | Pause |
| autonomous | Proceed（若無衝突）|

Override 規則：
- 3+ 個檔案衝突 → 無論 gate 強制暫停
- 任何 worktree base commit 與 HEAD 差距 > 20 commits → 暫停並警告

**Gate 呈現格式（After INTEGRATE）：**

```
## Gate: Integration Review

**Mission**: {任務描述}
**Worktrees merged**: {N}/{M}

### Merge Results
| Agent | Isolation | Files | Conflicts | Status |
|-------|-----------|-------|-----------|--------|
| Delta | worktree  | 12    | 1         | ✅ Resolved |
| Echo  | worktree  | 8     | 0         | ✅ Clean |
| Alpha | boundary  | 7     | 0         | ✅ Verified |

### Conflict Resolutions
- useSettingsStore.ts: 合併 Delta 新增的 2 欄位到現有 store

### Worktree Cleanup
- ✅ agent-delta removed
- ✅ agent-echo removed
- ✅ git worktree prune complete

Proceed to VERIFY? (yes / no / revert)
```

### 2. Worktree 管理改進（P0 — Critical）

**影響的檔案：**
- `commands/squad.md` — EXECUTE 階段
- `skills/mission-planning/SKILL.md` — 新增隔離策略

#### 2a. 強制從 HEAD 建立分支

在 EXECUTE 階段，spawning worktree agent 時：
- 明確指示 agent 從 `HEAD` 建立 worktree 分支
- 記錄 HEAD commit SHA 到戰鬥計畫 metadata
- INTEGRATE 階段用此 SHA 作為 merge base reference

新增到 persona prompt 的作業規範：
```
- 建立 worktree 時必須從 HEAD 分支，不可從其他 commit
- 使用 git diff HEAD（而非 git diff main）檢視自己的變更
```

#### 2b. 隔離策略矩陣

在 mission-planning 中新增 Step 5.5：

```
Step 5.5: 決定隔離策略

對每個 squad member，依據其任務性質選擇隔離方式：

| 策略 | 適用情境 | 範例 |
|------|---------|------|
| worktree | 修改程式碼，且所改檔案無其他 agent 觸及 | Delta（新功能開發） |
| file-boundary | 多個 agent 在主工作區作業，各有排他性的檔案所有權 | Alpha（UI 元件 A-F）、Bravo（UI 元件 G-M） |
| none | 只讀取程式碼或產出獨立文件 | Foxtrot（設計文件）、Golf（使用手冊） |

在戰鬥計畫輸出中為每個 agent 標記隔離方式。
```

#### 2c. Lint 忽略模式

在 VERIFY 階段和 integration skill 中：

```
執行 lint 時的標準作法：
1. 偵測 .claude/ 目錄是否存在
2. ESLint → --ignore-pattern ".claude/"
3. Biome → 檢查 biome.json 是否排除 .claude/
4. 其他 linter → 依工具語法附加排除規則
5. 在 RETRO 建議使用者永久加入專案 lint 設定
```

### 3. Agent 輸出標準化（P0 — Critical）

**變更：** 每個 agent 將成果寫到標準化目錄，不再依賴 in-memory 通訊

**新增到 `skills/role-forging/SKILL.md`：**

在每個 persona 的「作業規範」中加入輸出規範：

```
作業規範（追加）：
6. 完成所有 tasks 後，在 .claude/squad/outputs/{你的代號}/ 建立：
   - manifest.md：完成狀態、修改的檔案清單、新建的檔案清單
   - interface-changes.md：對共享檔案的結構性變更
     （新增的 exports、types、fields、functions）
```

**manifest.md 格式：**

```markdown
# Agent Output: {Callsign}
Status: complete | partial | blocked
Completed tasks: #1, #3
Blocked tasks: (none)

## Artifacts
- [code] src/engines/practice/FingeringEngine.ts — 指法演算法引擎
- [test] src/engines/practice/FingeringEngine.test.ts — 21 個測試
- [code] src/i18n/locales/en.ts — 英文翻譯檔
- [code] src/i18n/locales/zh-TW.ts — 繁中翻譯檔
- [spec] interface-changes.md — store 介面變更規格

## Modified Files
- src/stores/useSettingsStore.ts (shared — see interface-changes.md)

## New Files
- src/engines/practice/FingeringEngine.ts
- src/engines/practice/FingeringEngine.test.ts
- src/i18n/locales/en.ts
- src/i18n/locales/zh-TW.ts
```

**Artifact 類型標記**（借鏡 VibeHQ Artifact Registry）：

每個 artifact 必須標記類型，讓 INTEGRATE 和 DEBRIEF 可以分類處理：

| 類型 | 用途 |
|------|------|
| `[code]` | 程式碼檔案（新建或修改） |
| `[test]` | 測試檔案 |
| `[spec]` | 規格/介面變更文件 |
| `[doc]` | 使用者文件或開發者文件 |
| `[config]` | 設定檔（yaml, json, env 等） |

**完成必須附帶 artifact**（借鏡 VibeHQ Task Lifecycle）：

Agent 不能只回報「完成了」——每個 task 完成時，manifest.md 的 Artifacts 區塊必須列出該 task 產出的具體檔案。沒有 artifact 的 task 不算完成。INTEGRATE 階段會驗證 manifest 中列出的檔案是否實際存在。

**interface-changes.md 格式：**

```markdown
# Interface Changes: {Callsign}

## useSettingsStore.ts
### New types
- `Language = "en" | "zh-TW"`

### New fields
- `showFingering: boolean` (default: true)
- `language: Language` (default: detectLanguage())

### New actions
- `setShowFingering(show: boolean): void`
- `setLanguage(lang: Language): void`

### New exports
- `type Language`

## index.ts (barrel export)
### New exports
- `export { FingeringEngine } from './practice/FingeringEngine'`
```

**為什麼這是 P0：**
- 解決 context compaction 問題——成果持久化在檔案系統
- 讓 INTEGRATE 階段有結構化資料可讀
- Agent 斷線不影響成果收集

### 4. 共享狀態介面契約（P1 — High）

**影響的檔案：**
- `skills/mission-planning/SKILL.md` — 新增 Step 2.5
- `skills/role-forging/SKILL.md` — 共享檔案規範

#### 4a. 衝突風險評估（mission-planning）

在 Step 2 (Map Dependencies) 之後新增：

```
Step 2.5: 評估共享狀態衝突風險

1. 列出所有 agent 任務涉及的檔案
2. 找出被多個 agent 觸及的「共享檔案」
3. 對每個共享檔案標記存取模式：
   - read-only：只需要讀取不修改
   - additive：需要新增欄位/exports（相容性變更）
   - mutative：需要修改現有結構（破壞性變更）
4. 衝突矩陣：
   - 兩個 agent 都是 read-only → 無風險
   - 一個 additive + 一個 read-only → 低風險
   - 兩個都是 additive → 中風險（需要 interface contract）
   - 任何 mutative → 高風險（考慮合併到同一 agent）
```

#### 4b. 契約確認機制（借鏡 VibeHQ Contract System）

VibeHQ 的 contract 系統要求 agent 在開工前明確「簽核」規格文件。Squad 無法用 WebSocket 強制阻擋，但可以在 prompt 層面實現同等效果：

每個 agent 開工前必須在 outputs 目錄寫一份 `contract-ack.md`，確認理解任務邊界：

```
作業規範（追加）：
7. 開工前（寫任何程式碼之前），在 .claude/squad/outputs/{你的代號}/contract-ack.md 寫下：
   - 你理解的任務範圍（列出 task 編號和簡述）
   - 你會修改的檔案清單
   - 共享檔案的存取權限確認（read-only / additive）
   - 任何你認為不明確需要確認的事項
   這是你與團隊的契約確認——寫完後才開始作業
```

INTEGRATE 階段在 Phase 1（清點）時會讀取每個 agent 的 `contract-ack.md`，比對實際修改的檔案是否在契約範圍內。超出契約的修改會被標記為越界警告。

#### 4c. 介面契約（role-forging）

在 persona prompt 中加入共享檔案的介面快照：

```
**相關共享檔案契約：**

useSettingsStore.ts:
  現有介面：SettingsState { showNoteLabels, volume, practiceMode,
    fontSize, showKeyLabels, keyLabelType, midiDeviceId,
    transpose, playbackSpeed }
  你的權限：additive（可新增欄位，不可修改/刪除現有欄位）
  你需要新增：showFingering: boolean, language: Language

  規則：
  - 不可重新建立此 store
  - 不可修改現有欄位的型別或預設值
  - 新增的欄位要同步加入 PersistedSettings interface
  - 在 interface-changes.md 中完整記錄你的變更
```

#### 4d. 禁止事項追加

在每個 worktree agent 的禁止事項加入：

```
禁止事項（追加）：
- 不重新建立已存在的 store/config/type 檔案
- 修改共享檔案時只能 additive（新增），不可 mutative（修改/刪除）
- 修改共享檔案後必須在 interface-changes.md 記錄完整變更
```

### 5. Convoy 分批部署模式（P1 — High）

**影響的檔案：**
- `commands/squad.md` — EXECUTE 階段
- `skills/mission-planning/SKILL.md` — 部署策略

**規則：**
- 3-4 agents → 可選 convoy 或全部署
- 5+ agents → **必須**使用 convoy 模式
- Convoy 按 dependency wave 分批

**EXECUTE 階段改寫：**

```
3. EXECUTE — 團隊部署

如果 squad size ≥ 5 或計畫包含多個 dependency wave：
  使用 Convoy 模式：

  3a. 部署 Wave 1 agents
  3b. 監控 Wave 1 直到所有 agent 完成
  3c. 收集 Wave 1 outputs（讀取 .claude/squad/outputs/）
  3d. 微整合（如果 Wave 2 agents 需要 Wave 1 的介面變更資訊）
  3e. 部署 Wave 2 agents（persona prompt 中包含 Wave 1 的 interface-changes）
  3f. 監控 Wave 2 直到所有 agent 完成
  3g. 重複直到所有 waves 完成

否則：
  一次部署所有 agents，監控直到完成
```

**訊息傳遞狀態檢查**（借鏡 VibeHQ Idle-Aware Message Queue）：

VibeHQ 的 Hub 會在 agent 忙碌時佇列訊息，idle 時才 flush。Squad 沒有訊息佇列 server，但可以在監控邏輯中加入類似的狀態感知：

```
監控規則（EXECUTE 階段）：
- 在向 agent 發送非緊急訊息前，先檢查其 outputs 目錄狀態：
  - 有 .complete → agent 已完成，不需要發訊息
  - 有 contract-ack.md 但無 .complete → agent 正在作業中
  - 無 contract-ack.md → agent 可能還在啟動
- 僅在以下情況主動發送訊息：
  - agent 回報 blocker（必須立即回應）
  - agent 超時未回應（需要確認狀態）
  - 有新的 Wave 成果資訊需要傳遞（convoy 模式）
- 避免在 agent 正常作業中發送不必要的狀態查詢——減少 context 干擾
```

**好處：**
- 減少同時活躍 agent 數量 → 降低 context 壓力
- Wave 2 agents 獲得 Wave 1 的成果資訊 → 減少衝突
- Wave 1 失敗時不浪費 Wave 2 的計算資源
- 訊息狀態感知 → 降低不必要的 context 干擾

### 6. 前置檢查與回滾機制（P1 — High）

**影響的檔案：**
- `commands/squad.md` — PLAN 和 INTEGRATE 之間

#### 6a. Pre-flight Checks（PLAN 結束、EXECUTE 開始之前）

```
Pre-flight:
1. 記錄 HEAD commit SHA 作為 baseline
2. 執行 lint → 記錄基線狀態（pass/fail + error count）
3. 執行 typecheck → 記錄基線
4. 執行 test → 記錄基線（test count、pass count）
5. 將基線寫入 .claude/squad/state/baseline.md
```

目的：VERIFY 階段可以比對「之前就有的問題」vs「agent 引入的新問題」。

#### 6b. Rollback 策略（INTEGRATE 失敗時）

```
如果 INTEGRATE Phase 3 合併過程中發生錯誤：
1. 還原主工作區：git checkout -- .
2. 保留 worktrees 不清理（讓使用者或後續 agent 可以重試）
3. 報告失敗原因和建議的手動修復步驟
4. 暫停等待使用者決定（無論 gate level）
```

### 7. Context Compaction 韌性（P1 — High）

**影響的檔案：**
- `commands/squad.md` — EXECUTE、DEBRIEF
- `skills/role-forging/SKILL.md` — persona template

#### 7a. Agent 完成標記

Agent 完成所有 tasks 後，除了寫 `manifest.md`，還要：
- 在 outputs 目錄建立 `.complete` 標記檔案
- 參謀總長定期掃描 `.claude/squad/outputs/*/`.complete` 判斷完成狀態
- 不再純依賴 SendMessage/TaskOutput 確認完成

#### 7b. Agent 失聯偵測與重生（借鏡 VibeHQ Hot Respawn）

VibeHQ 透過 spawner subscription 機制，在 agent 斷線時自動重新連接。Squad 沒有持久化 server，但可以用檔案系統實現同等效果：

在 EXECUTE 監控邏輯中加入失聯偵測和重生機制：

```
Agent 失聯處理：
1. 如果 TaskOutput 超時或 agent 長時間無回應：
   a. 檢查 outputs 目錄：
      - 有 .complete → agent 已完成，無需處理
      - 有 manifest.md 但無 .complete → 部分完成
      - 無任何 outputs → agent 可能啟動失敗
2. 對「部分完成」或「啟動失敗」的 agent：
   a. 讀取 manifest.md（如有）判斷已完成的 tasks
   b. 建立 respawn persona：
      - 包含原始 persona 全部內容
      - 追加：「以下 tasks 已由前任完成，你只需處理剩餘 tasks」
      - 列出已完成的 tasks 和對應 artifacts
   c. Spawn 新 agent 繼續未完成的 tasks
   d. 記錄 respawn 事件到 .claude/squad/state/
3. Respawn 限制：
   - 同一個 task 最多 respawn 1 次
   - 如果 respawn 後仍然失聯 → 暫停等待使用者決定
```

#### 7c. 容錯團隊清理

DEBRIEF 階段的 TeamDelete 改為容錯流程：

```
團隊清理：
1. 嘗試 SendMessage shutdown_request 到每個 member
2. 等待 30 秒（或 member 回應）
3. 嘗試 TeamDelete
4. 如果 TeamDelete 失敗（active members 錯誤）：
   a. 記錄警告：「{N} 個 agent 未正常關閉，強制清理」
   b. 手動刪除團隊狀態檔案
   c. 繼續 pipeline（不阻塞）
5. 在 RETRO 中記錄 compaction event
```

### 8. /squad --abort 中止機制（P2 — Medium）

**影響的檔案：**
- `commands/squad.md` — 子命令區塊

新增子命令：

```
--abort [--keep-worktrees]
  中止當前進行中的任務。

  1. 讀取 .claude/squad/state/ 判斷當前階段和活躍 agents
  2. 嘗試 SendMessage abort 到所有活躍 agents
  3. 容錯 TeamDelete（同改進 7b）
  4. 如果未指定 --keep-worktrees：清理所有 agent worktrees
  5. 寫入中止報告到 .claude/squad/reports/YYYY-MM-DD-aborted-{slug}.md
  6. 清理 .claude/squad/state/ 和 .claude/squad/outputs/
```

### 9. VERIFY 階段改進（P2 — Medium）

**影響的檔案：**
- `commands/squad.md` — VERIFY 區塊

#### 9a. 驗證順序

改為嚴格順序執行：

```
1. Lint（最快修復，先跑先修）
   → 如果失敗且可自動修復 → 修復後重跑
2. Typecheck（型別問題在測試前抓到省時間）
   → 如果失敗 → 修復後重跑
3. Test（最慢但最全面）
   → 如果失敗 → 分析是新引入的還是基線就有的（對比 pre-flight baseline）
```

#### 9b. 與基線的差異分析

```
驗證結果報告：
- Lint: 2 errors（基線: 0 → 新增 2 個錯誤 ← 需修復）
- Typecheck: pass（基線: pass → 無退化）
```

## 設計核心洞察

> Squad v0.1 的設計假設 agent 是無狀態的獨立個體，可以自由平行執行後由 VERIFY 階段驗收。但實戰證明：**agent 的成果需要持久化、共享狀態需要契約化、部署需要分批化、合併需要專門的階段處理。**
>
> v0.2 的核心改變是引入四個機制來解決這個根本矛盾：
> 1. **Agent 輸出標準化** — 成果不再是 ephemeral 的 SendMessage，而是持久化在檔案系統，附帶類型標記
> 2. **契約確認模式** — 共享可變狀態由 PLAN 階段建立契約，agent 開工前明確簽認，只能做 additive 變更
> 3. **INTEGRATE 階段** — 一個具備全局視角的合併節點，驗證 artifact 存在性和契約遵守情況
> 4. **韌性機制** — 失聯偵測 + respawn、容錯清理、訊息狀態感知，降低 context compaction 的影響
